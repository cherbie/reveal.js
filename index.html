Concurrency vs Parallelism

Concurrency

Concurrency concept that allows multiple computations to happen simultaneously
This is achieved by dividing a program into smaller, independent tasks that can be executed in parallel.
This approach can significantly improve the performance of your programs, especially on multi-core processors.

Parallelism

  • Parallelism is the concept of doing multiple things at the same time
  • Parallelism is the simultaneous physical execution of tasks at runtime, and it requires hardware with multiple computing resources. It resides on the hardware layer.
  • Parallelism is a subclass of concurrency.

Multithreading

Threads are a way to achieve parallelism.

The goal is to maximize the utilization of CPU cores and to improve the overall performance of the program.

  • A thread is the smallest sequence of programmed instructions that can be managed independently by a scheduler.
  • Threads within a process share the same data space, which allows them to communicate with each other more easily than if they were separate processes.
  • Managing threads effectively is crucial for achieving the benefits of assembly concurrency.
  • Each thread has its own set of registers, stack, and program counter.
  • The kernel / operating systems job is to manage these threads, decide when to pause and resume each one and ensure they don't interfere with each other

Synchronization

Without proper synchronization, threads could access and modify shared data simultaneously, leading to inconsistencies and errors.

  • Synchronization is the coordination of threads to ensure that they work together properly.
  • "Race Conditions"
  • Very difficult debugging
  • We need to mitigate these risks
  • Synchronization can be achieved through various mechanisms such as locks, semaphores, and condition variables.

Concurrency Primitives

Assembly

						
section .data
semaphore db 1

section .text
global _start
_start:
	mov ecx, 5
create_thread:
	push ecx
	mov eax, 2 ; sys_fork
	int 0x80
	pop ecx
	test eax, eax
	jz child_process
	loop create_thread
	jmp exit
child_process:
	wait_semaphore:
		cmp byte [semaphore], 0
		je wait_semaphore
	dec byte [semaphore]
	inc byte [counter]
	inc byte [semaphore]
exit:
	mov eax, 1 ; sys_exit
	int 0x80

Atomics

- the operation happens as a single indivisible operation, not as multiple parts which can happen partially before and partially after something else - atomic operations are more efficient than mutexes for simple operations like increments and decrements - directly manipulate memory without locking - You get atomicity "for free" with no extra hardware for aligned loads or stores up to the size of the data paths between cores, memory, and I/O busses - temporary blocking of other requests is needed to keep an atomic transfer atomic
